# 设备组状态同步机制分析

## 当前实现流程

```
用户点击开关 → toggleAllUnits()
    ↓
1. 本地状态更新（创建新对象）
    ↓
2. 发送API请求 → ApiService.controlUnit()
    ↓
3. 等待WebSocket推送 → StateUpdateMessage
    ↓
4. handleStateUpdate() → 更新设备组状态
    ↓
5. 设备组颜色重新计算 → getGroupStatusColor()
```

## 存在的问题

### ❌ 问题1：状态更新延迟
- API请求异步执行，本地状态更新和API调用分离
- 依赖WebSocket推送，存在网络延迟
- 用户看到的状态更新可能滞后

### ❌ 问题2：数据不一致风险
- 本地状态和服务器状态可能暂时不一致
- 如果WebSocket未推送，设备组颜色不会更新
- 没有失败重试机制

### ❌ 问题3：对象创建开销
- 每次都创建新的DeviceUnit对象
- V2装饰器虽然能检测变化，但频繁创建对象影响性能

## 状态同步架构图

```
┌─────────────────────────────────────────────────────────┐
│                    Index 页面                           │
│  ┌──────────────────────────────────────────────────┐  │
│  │  toggleAllUnits()                                 │  │
│  │  1. 计算新状态                                     │  │
│  │  2. 直接修改 @Trace 属性（立即触发UI更新）         │  │
│  │  3. 发送API请求                                   │  │
│  └──────────────────────────────────────────────────┘  │
│                         ↓                               │
│  ┌──────────────────────────────────────────────────┐  │
│  │  @Trace deviceGroups (自动检测变化)               │  │
│  │  - units[].status 变化 → 触发UI更新               │  │
│  │  - getGroupStatusColor() 重新计算                 │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                  WebSocket 推送（异步）                  │
│  StateUpdateMessage → handleStateUpdate()               │
│  - 二次确认服务器状态                                    │
│  - 处理其他客户端的状态变更                               │
└─────────────────────────────────────────────────────────┘
```
